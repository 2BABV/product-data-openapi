## Schema Structure Overview

The ETIM xChange schema is a **catalog exchange format** designed for file-based product data exchange. It uses a hierarchical structure:

```
Root (Catalogue)
├── Catalogue metadata (ID, type, validity, language, currency)
├── Supplier[] (1..n)
│   ├── Supplier identity (GLN, DUNS, name)
│   ├── SupplierAttachments[]
│   └── Product[] (1..n)
│       ├── ProductIdentification (required)
│       ├── ProductDetails (includes ProductDescriptions[])
│       ├── ProductRelations[]
│       ├── Legislation
│       ├── LcaEnvironmental
│       ├── ProductAttachments[]
│       ├── EtimClassification[]
│       ├── OtherClassifications[]
│       ├── ProductCountrySpecificFields[]
│       ├── ProductCountrySpecificExtensions[]
│       └── TradeItem[] (0..n)
│           ├── ItemIdentification (required)
│           ├── ItemDetails (includes ItemDescriptions[])
│           ├── ItemRelations[]
│           ├── ItemLogisticDetails[]
│           ├── Ordering (required)
│           ├── Pricing[] (required, includes AllowanceSurcharge[])
│           ├── ItemAttachments[]
│           ├── ItemCountrySpecificFields[]
│           ├── ItemCountrySpecificExtensions[]
│           ├── PackagingUnit[]
│           └── TradeItemEnclosed[]
```

## Key Concepts for API Mapping

### Naming Convention Translation
ETIM uses **PascalCase** field names. The OpenAPI APIs use **camelCase** properties:
- `ManufacturerIdGln` → `manufacturerIdGln`
- `ProductDescriptions` → `productDescriptions`
- `EtimFeatureCode` → `etimFeatureCode`

### Type Conversions (ETIM → OpenAPI)
ETIM stores decimal numbers as regex-validated strings because it was designed for catalog file exchange (BMEcat/XML). The API converts these to proper numeric types:

| ETIM Pattern | OpenAPI Type | Constraints |
|---|---|---|
| `^[0-9]{1,11}[.]{0,1}[0-9]{0,4}$` | `number` | `format: decimal`, `minimum: 0`, `multipleOf: 0.0001`, `maximum: 99999999999.9999` |
| `^[0-9]{1,5}[.]{0,1}[0-9]{0,4}$` | `number` | `format: decimal`, `minimum: 0`, `multipleOf: 0.0001`, `maximum: 99999.9999` |
| `^[0-9]{1,12}[.]{0,1}[0-9]{0,4}$` | `number` | `format: decimal`, `minimum: 0`, `multipleOf: 0.0001`, `maximum: 999999999999.9999` |
| `^[-]{0,1}[0-9]{1,12}[.]{0,1}[0-9]{0,4}$` | `number` | `format: decimal`, `multipleOf: 0.0001`, `minimum: -999999999999.9999`, `maximum: 999999999999.9999` |

Other key conversions:
- ETIM `integer` with `minimum`/`maximum` → OpenAPI `integer` (preserve as-is)
- ETIM `boolean` → OpenAPI `boolean` (preserve as-is)
- ETIM multi-state string enums like `"true" | "false" | "exempt"` → OpenAPI `string` with `enum` (do NOT convert to boolean)
- ETIM `format: date` → OpenAPI `format: date` with ISO 8601 description
- ETIM `format: email` → OpenAPI `format: email`
- ETIM `format: uri` → OpenAPI `format: uri`

### Catalog-Level Field Denormalization
Some fields in ETIM are catalog-level (root object) but need to be denormalized into API responses:
- `CurrencyCode` → required `currencyCode` on pricing records
- `Language[]` → used for language filtering parameters

### ETIM Composite Keys
- **Products**: `ManufacturerIdGln` + `ManufacturerProductNumber`
- **Trade Items**: `SupplierIdGln` (from Supplier) + `SupplierItemNumber` (from ItemIdentification)

### Field Traceability Convention
Every OpenAPI property mapped from ETIM must include traceability in its description:
```yaml
description: |
  [Business meaning]

  **ETIM xChange**: `FieldName`
  **Path**: `Supplier[].Product[].Section.FieldName`
```

## Comparing Schema Versions

When a new ETIM xChange schema version is released:
1. Place the new JSON schema file in this directory
2. Use the `compare-etim-schema-versions` prompt to diff against the current version
3. Focus on: new fields, removed fields, type changes, enum value additions/removals, pattern changes
4. Check the mapping registry at `etim-field-mapping-registry.md` to identify which API schemas are affected
5. Update the API schemas, mapping registry, and this `.instructions.md` with the new version info
