# Product API — AI Context

## Overview

This is the Product API (v1.0.0-Preview1) based on ETIM xChange V2.0. Products are identified by composite key: `manufacturerIdGln` (GLN, 13 digits) + `manufacturerProductNumber` (max 35 chars). The API exposes product data as a set of aspect-based endpoints (details, descriptions, classifications, LCA environmental, attachments) with both single-item and bulk variants. Bulk endpoints use cursor-based pagination via `CursorPaginationMetadata`.

## Current Endpoints

### Single-item endpoints

| Path | operationId | Response schema | Domain data |
|------|-------------|-----------------|-------------|
| `GET /products/{manufacturerIdGln}/{manufacturerProductNumber}` | `getProduct` | `ProductResponse` → `ProductResponseData` | Full `Product` (all nested sections) |
| `GET /products/{manufacturerIdGln}/{manufacturerProductNumber}/details` | `getProductDetails` | `ProductDetailsResponse` → `ProductDetailsResponseData` | `ProductDetails` |
| `GET /products/{manufacturerIdGln}/{manufacturerProductNumber}/descriptions` | `getProductDescriptions` | `ProductDescriptionsResponse` → `ProductDescriptionsResponseData` | `ProductDescription[]` |
| `GET /products/{manufacturerIdGln}/{manufacturerProductNumber}/etim-classifications` | `getProductEtimClassifications` | `ProductEtimClassificationsResponse` → `ProductEtimClassificationsResponseData` | `EtimClassification[]` (with `EtimFeature[]`) |
| `GET /products/{manufacturerIdGln}/{manufacturerProductNumber}/lca-environmental` | `getProductLcaEnvironmental` | `ProductLcaEnvironmentalResponse` → `ProductLcaEnvironmentalResponseData` | `LcaEnvironmental` (with `LcaDeclaration[]`) |
| `GET /products/{manufacturerIdGln}/{manufacturerProductNumber}/attachments` | `getProductAttachments` | `ProductAttachmentsResponse` → `ProductAttachmentsResponseData` | `ProductAttachment[]` (with `AttachmentDetails[]`) |

### Bulk endpoints

| Path | operationId | Response schema | Items schema |
|------|-------------|-----------------|--------------|
| `GET /products/bulk/details` | `getBulkProductDetails` | `BulkProductDetailsResponse` | `ProductDetailsSummary` |
| `GET /products/bulk/descriptions` | `getBulkProductDescriptions` | `BulkProductDescriptionsResponse` | `ProductDescriptionsSummary` |
| `GET /products/bulk/etim-classifications` | `getBulkProductEtimClassificationFeatures` | `BulkProductEtimClassificationFeaturesResponse` | `ProductEtimClassificationFeature` |
| `GET /products/bulk/lca-environmental` | `getBulkProductLcaDeclarations` | `BulkProductLcaDeclarationsResponse` | `ProductLcaDeclarationSummary` |
| `GET /products/bulk/attachments` | `getBulkProductAttachments` | `BulkProductAttachmentsResponse` | `ProductAttachmentSummary` |

All bulk endpoints accept query parameters: `cursor`, `limit`, `selectionId`, `manufacturerIdGln` (filter), `mutationDateTime`. Some also accept `language`, `etimReleaseVersion`, or `lifeCycleStage`.

> **Not wired in `openapi.yaml` paths** (path files exist but are not referenced): `bulk/products.yaml` (`getBulkProducts`), `bulk/product-etim-classifications.yaml` (`getBulkProductEtimClassifications`).

## Schema Architecture

### Domain schemas (`schemas/domain/`)

| File | Purpose | Key properties |
|------|---------|----------------|
| `Product.yaml` | Full product aggregate with all nested sections | `manufacturerIdGln`, `manufacturerProductNumber`, `productDescriptions[]`, `etimClassifications[]`, `legislation`, `lcaEnvironmental`, `productAttachments[]`, `productRelations[]`, `productCountrySpecificFields[]` |
| `ProductDetails.yaml` | Core product info (identification + operational details) without composite key | `productGtins[]`, `brandName`, `unbrandedProduct`, `productValidityDate`, `productStatus`, `productType`, `customsCommodityCode`, `countryOfOrigin[]`, `warrantyConsumer`, `warrantyBusiness`, `relatedManufacturerProductGroup[]` |
| `ProductDetailsSummary.yaml` | Flattened product details for bulk retrieval (includes composite key) | Same as `ProductDetails` plus `manufacturerIdGln`, `manufacturerProductNumber` |
| `ProductDescription.yaml` | Multilingual product descriptions (one object per language) | `descriptionLanguage`, `minimalProductDescription`, `uniqueMainProductDescription`, `fullProductDescription`, `productMarketingText`, `productSpecificationText`, `productApplicationInstructions`, `productKeyword[]`, `productPageUrl` |
| `ProductDescriptionsSummary.yaml` | Flattened description for bulk (includes composite key + all 6 text fields) | Composite key + all `ProductDescription` fields |
| `EtimClassification.yaml` | ETIM class with features | `etimClassCode`, `etimClassVersion`, `etimReleaseVersion`, `etimDynamicReleaseDate`, `etimFeatures[]` |
| `EtimFeature.yaml` | Single ETIM feature value (alphanumeric, numeric, range, logical) | `etimFeatureCode`, `etimValueCode`, `etimValueNumeric`, `etimValueRangeLower`, `etimValueRangeUpper`, `etimValueLogical`, `etimValueDetails[]`, `reasonNoValue` |
| `ProductEtimClassificationFeature.yaml` | Flattened feature row for bulk (one row per feature per class per product) | Composite key + classification fields + all feature value fields |
| `ProductEtimClassificationsSummary.yaml` | Product ETIM classification for bulk (includes nested `EtimClassification[]`) | Composite key + `etimClassifications[]` |
| `LcaEnvironmental.yaml` | LCA/EPD data container | `declaredUnitUnit`, `declaredUnitQuantity`, `functionalUnitDescription[]`, `lcaReferenceLifetime`, `thirdPartyVerification`, EPD dates/IDs, PCR/PSR descriptions, `lcaDeclaration[]` |
| `LcaDeclaration.yaml` | Single life-cycle stage declaration with 22 environmental indicators | `lifeCycleStage` (A1–D), `lcaDeclarationIndicator`, GWP (total/fossil/biogenic/LULUC), AP, EP (freshwater/marine/terrestrial), ODP, POCP, ADP (elements/fossil), WDP, PERT, PENRT, PM, IRP, ETP-fw, HTP-c, HTP-nc, SQP |
| `ProductLcaDeclarationSummary.yaml` | Flattened LCA declaration for bulk (includes parent LCA context) | Composite key + parent LCA fields + all declaration indicators |
| `ProductAttachment.yaml` | Attachment container with type and detail files | `attachmentType`, `attachmentTypeSpecification`, `attachmentOrder`, `attachmentDetails[]` |
| `AttachmentDetails.yaml` | Single attachment file entry | `attachmentUri`, `attachmentFilename`, `attachmentLanguage[]`, `attachmentDescription[]`, `attachmentIssueDate`, `attachmentExpiryDate` |
| `ProductAttachmentSummary.yaml` | Flattened attachment for bulk (one row per file, denormalized type) | Composite key + `attachmentType`, `attachmentTypeSpecification`, `attachmentOrder`, `attachmentUri`, `attachmentFilename`, `attachmentIssueDate`, `attachmentExpiryDate` |
| `Legislation.yaml` | Legal compliance data (CE, RoHS, REACH, WEEE, hazmat) | `electricComponentContained`, `batteryContained`, `weeeCategory`, `rohsIndicator`, `rohsExemptionNumber`, `ceMarking`, `sdsIndicator`, `reachIndicator`, `reachDate`, `scipNumber`, `ufiCode`, `unNumber`, `hazardClass[]`, `adrCategory`, `netWeightHazardousSubstances`, `volumeHazardousSubstances` |
| `ProductRelation.yaml` | Product relationship (accessory, sparepart, etc.) | `relationType`, `relatedProducts[]` |
| `RelatedProduct.yaml` | Related product reference | `relatedManufacturerProductNumber`, `relatedProductGtin[]`, `relatedProductQuantity` |
| `ProductCountrySpecificField.yaml` | Country-specific key-value fields (simplified model) | `countryCode`, `fieldName`, `fieldValue` |

### Response schemas (`schemas/responses/`)

All responses follow the `data` envelope pattern:
- **Single-item**: `{ data: $ref: ./XxxResponseData.yaml }` — the `ResponseData` schema contains the composite key + a `$ref` to the domain schema.
- **Bulk**: `{ data: [ items: $ref: ./XxxSummary.yaml ], meta: $ref: CursorPaginationMetadata }`.

Response files: `ProductResponse`, `ProductDetailsResponse`, `ProductDescriptionsResponse`, `ProductEtimClassificationsResponse`, `ProductLcaEnvironmentalResponse`, `ProductAttachmentsResponse`, and their `*ResponseData` counterparts. Bulk responses: `BulkProductDetailsResponse`, `BulkProductDescriptionsResponse`, `BulkProductEtimClassificationFeaturesResponse`, `BulkProductLcaDeclarationsResponse`, `BulkProductAttachmentsResponse`.

### Enum schemas (`schemas/enums/`)

| File | Values | Used by |
|------|--------|---------|
| `ProductStatus.yaml` | `PRE-LAUNCH`, `ACTIVE`, `ON HOLD`, `PLANNED WITHDRAWAL`, `OBSOLETE` (nullable) | `ProductDetails`, `ProductDetailsSummary`, `Product` |
| `ProductType.yaml` | `PHYSICAL`, `CONTRACT`, `LICENCE`, `SERVICE` (nullable) | `ProductDetails`, `ProductDetailsSummary`, `Product` |
| `AttachmentType.yaml` | 22 codes: `ATX001`–`ATX026`, `ATX099` (images, docs, CAD, BIM, etc.) | `ProductAttachment`, `ProductAttachmentSummary` |
| `AttachmentTypeSpecification.yaml` | 60+ codes: `MDX002`–`MDX037` and more (specification subtypes) (nullable) | `ProductAttachment`, `ProductAttachmentSummary` |
| `WeeeCategory.yaml` | `1`–`6` (nullable) | `Legislation` |
| `ProductRelationType.yaml` | 15 values: `ACCESSORY`, `CONSISTS_OF`, `CONSUMABLES`, `CROSS-SELLING`, `MAIN_PRODUCT`, `MANDATORY`, `PREDECESSOR`, `REFURBISHED`, `SELECT`, `SIMILAR`, `SPAREPART`, `SUCCESSOR`, `UPSELLING`, `USED`, `OTHER` | `ProductRelation` |

## ETIM xChange V2.0 Mapping Status

### Fully mapped ETIM sections

| ETIM xChange section | API schema(s) | Notes |
|----------------------|---------------|-------|
| `ProductIdentification` (core fields) | `ProductDetails.yaml`, `ProductDetailsSummary.yaml`, `Product.yaml` | `ManufacturerProductNumber`, `ManufacturerIdGln`, `ProductGtin[]`, `UnbrandedProduct`, `BrandName`, lifecycle dates, `CustomsCommodityCode`, `FactorCustomsCommodityCode`, `CountryOfOrigin[]` |
| `ProductDetails` (operational) | `ProductDetails.yaml`, `ProductDetailsSummary.yaml`, `Product.yaml` | `ProductStatus`, `ProductType`, `CustomisableProduct`, `HasSerialNumber`, `WarrantyConsumer`, `WarrantyBusiness`, `RelatedManufacturerProductGroup[]` |
| `ProductDetails.ProductDescriptions[]` | `ProductDescription.yaml`, `ProductDescriptionsSummary.yaml` | All 6 text fields + keywords + URL: `MinimalProductDescription`, `UniqueMainProductDescription`, `FullProductDescription`, `ProductMarketingText`, `ProductSpecificationText`, `ProductApplicationInstructions`, `ProductKeyword[]`, `ProductPageUrl` |
| `EtimClassification[]` (core + features) | `EtimClassification.yaml`, `EtimFeature.yaml`, `ProductEtimClassificationFeature.yaml` | `EtimClassCode`, `EtimClassVersion`, `EtimReleaseVersion`, `EtimDynamicReleaseDate`, all feature value types + `ReasonNoValue` + `EtimValueDetails[]` |
| `LcaEnvironmental` | `LcaEnvironmental.yaml`, `LcaDeclaration.yaml`, `ProductLcaDeclarationSummary.yaml` | Full declared unit, EPD metadata, PCR/PSR, functional unit descriptions, all 22 EN 15804 environmental indicators across A1–D lifecycle stages |
| `ProductAttachments[]` | `ProductAttachment.yaml`, `AttachmentDetails.yaml`, `ProductAttachmentSummary.yaml` | All 22 attachment types + 60+ specification subtypes, all detail fields (URI, filename, language, description, dates) |
| `ProductRelations[]` | `ProductRelation.yaml`, `RelatedProduct.yaml` | All 15 relation types, `RelatedManufacturerProductNumber`, `RelatedProductGtin[]`, `RelatedProductQuantity` |
| `ProductCountrySpecificFields[]` (basic) | `ProductCountrySpecificField.yaml` | `CountryCode`, `FieldName`, `FieldValue` (simplified string model) |

### Partially mapped ETIM sections

- **Legislation**: ~44% mapped (16 of ~36 fields). Implemented: `ElectricComponentContained`, `BatteryContained`, `WeeeCategory`, `RohsIndicator`, `RohsExemptionNumber`, `CeMarking`, `SdsIndicator`, `ReachIndicator`, `ReachDate`, `ScipNumber`, `UfiCode`, `UnNumber`, `HazardClass[]`, `AdrCategory`, `NetWeightHazardousSubstances`, `VolumeHazardousSubstances`. **Missing**: `SvhcIdentification[]` (with `CasNumber`, `EcNumber`), GHS labeling (`LabelCode[]`, `SignalWord`, `HazardStatement[]`, `PrecautionaryStatement[]`), ADR transport (`PackingGroup`, `LimitedQuantities`, `ExceptedQuantities`, `AggregationState`, `SpecialProvisionId[]`, `ClassificationCode`, `HazardLabel[]`, `TunnelCode`, `EnvironmentalHazards`, `UnShippingName[]`), battery data (`LiIonTested`, `LithiumAmount`, `BatteryEnergy`, `Nos274`), other (`HazardTrigger[]`, `EprelRegistrationNumber`).
- **BrandDetails** (within `ProductIdentification`): `BrandName` mapped but `BrandSeries[]` and `BrandSeriesVariation[]` not yet implemented (multilingual nested arrays).
- **ProductCountrySpecificFields**: Basic `fieldName`/`fieldValue` string model supported but `CSProductCharacteristicValueUnitCode` and `CSProductCharacteristicReferenceGtin[]` not utilized. Full typed-value support (boolean, numeric, range, multilingual, single/multi select) not implemented.

### Intentionally excluded ETIM sections

- **OtherClassifications**: API focuses on ETIM classifications only; UNSPSC, eCl@ss, etc. are out of scope.
- **ManufacturerIdDuns, ManufacturerName, ManufacturerShortname**: Belong in a separate Manufacturer service; only `ManufacturerIdGln` is retained as a foreign key.
- **EtimModellingClassCode & EtimModellingPorts[]**: BIM/CAD modelling classes not yet in scope (TODO for future).
- **ProductCountrySpecificExtensions**: Open-ended extension point, not in scope.

## Design Decisions

- Product details endpoint merges ETIM's `ProductIdentification` + `ProductDetails` sections into a single response (same pattern as TradeItem details).
- Descriptions are separated from details into a dedicated `/descriptions` subresource with language filtering support.
- Bulk endpoints flatten nested structures for ETL/data warehouse ingestion (one row per language, one row per feature, one row per attachment file, one row per LCA declaration).
- LCA environmental data follows the full A1–D lifecycle stage model from EN 15804 with 22 environmental indicators.
- Product descriptions include all 6 ETIM text fields per language row plus keywords and page URL.
- Attachments support all 22 ETIM attachment types (`ATX001`–`ATX099`) plus 60+ specification subtypes (`MDX*`).
- `ProductLcaDeclarationSummary` excludes `functionalUnitDescription` (multilingual array) because it breaks the flat/tabular bulk pattern; consumers needing it should use the single-product `/lca-environmental` endpoint.
- Manufacturer identity (name, shortname, DUNS) deliberately excluded — use a separate Manufacturer service.

## Key Patterns

- All responses use the `data` envelope pattern (see `docs/envelope-pattern.md`).
- Single-item response: `data: $ref: ./XxxResponseData.yaml` — never use an inline anonymous object for `data`.
- Bulk response: `data: [array of items]` + `meta: CursorPaginationMetadata`.
- Domain schemas go in `schemas/domain/`, response envelopes in `schemas/responses/`.
- Enum files are separate when reused across schemas or have >5 values.
- Use `format: decimal` on all ETIM-converted number fields (code-gen hint for NSwag/.NET `decimal`).
- Use `type: ["string", "null"]` for nullable fields (OpenAPI 3.1 / JSON Schema 2020-12).
- Every property description includes ETIM xChange traceability: field name and full path.
- Use `examples` (plural array) in schemas, not `example` (singular, deprecated).

## When Adding New Endpoints or Fields

1. Check the ETIM xChange schema at `resources/etim-xchange/ETIM xChange_Schema_V2.0-2025-11-27.json`
2. Follow the naming conventions in the root copilot instructions (`.github/copilot-instructions.md`)
3. Add ETIM traceability to every new property description (`**ETIM xChange**: FieldName` + `**Path**: ...`)
4. Register all new schemas/parameters in `openapi.yaml` under `components`
5. Create `*Summary.yaml` variants for bulk endpoints (flattened with composite key)
6. Create `*ResponseData.yaml` for single-item responses (composite key + `$ref` to domain schema)
7. Regenerate the bundle after any source spec change:
   ```bash
   npx @redocly/cli bundle --config openapi/redocly.yaml product@v1 -o openapi/apis/product/generated/product-api.yaml
   ```
8. Update `CHANGELOG.md` with the change
