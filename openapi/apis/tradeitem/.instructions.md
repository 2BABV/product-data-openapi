# TradeItem API — AI Context

## Overview

This is the Trade Item API (v1.0.0-Preview1) based on ETIM xChange V2.0. Trade items are identified by composite key: `supplierIdGln` (GLN, 13 digits) + `supplierItemNumber` (max 35 chars). Each trade item references a parent product via `manufacturerIdGln` + `manufacturerProductNumber`, enabling joins with the Product API. The API exposes trade item data as aspect-based endpoints (details, descriptions, orderings, pricings, relations, logistics-details, attachments, enclosed-items) with both single-item and bulk variants. Bulk endpoints use cursor-based pagination via `CursorPaginationMetadata`.

## Current Endpoints

### Single-item endpoints

| Path | operationId | Response schema | Domain data |
|------|-------------|-----------------|-------------|
| `GET /trade-items/{supplierIdGln}/{supplierItemNumber}/details` | `getTradeItemDetails` | `TradeItemDetailsResponse` → `TradeItemDetailsResponseData` | `TradeItemDetails` (ItemIdentification + ItemDetails merged) |
| `GET /trade-items/{supplierIdGln}/{supplierItemNumber}/descriptions` | `getTradeItemDescriptions` | `TradeItemDescriptionsResponse` → `TradeItemDescriptionsResponseData` | `ItemDescription[]` |
| `GET /trade-items/{supplierIdGln}/{supplierItemNumber}/orderings` | `getTradeItemOrderings` | `TradeItemOrderingsResponse` → `TradeItemOrderingsResponseData` | `TradeItemOrdering` (single object, not array) |
| `GET /trade-items/{supplierIdGln}/{supplierItemNumber}/pricings` | `getTradeItemPricings` | `TradeItemPricingsResponse` → `TradeItemPricingsResponseData` | `TradeItemPricing[]` (with nested `AllowanceSurcharge[]`) |
| `GET /trade-items/{supplierIdGln}/{supplierItemNumber}/relations` | `getTradeItemRelations` | `TradeItemRelationsResponse` → `TradeItemRelationsResponseData` | `ItemRelation[]` |
| `GET /trade-items/{supplierIdGln}/{supplierItemNumber}/logistics-details` | `getTradeItemLogisticsDetails` | `TradeItemLogisticsDetailsResponse` → `TradeItemLogisticsDetailsResponseData` | `ItemLogistic[]` |
| `GET /trade-items/{supplierIdGln}/{supplierItemNumber}/attachments` | `getTradeItemAttachments` | `TradeItemAttachmentsResponse` → `TradeItemAttachmentsResponseData` | `ItemAttachment[]` |
| `GET /trade-items/{supplierIdGln}/{supplierItemNumber}/enclosed-items` | `getTradeItemEnclosedItems` | `TradeItemEnclosedItemsResponse` → `TradeItemEnclosedItemsResponseData` | `TradeItemEnclosedItem[]` |

> **Domain-only endpoint** (in `openapi-domain.yaml`, not in the main API): `getTradeItem` returns `TradeItemResponse` → `TradeItemResponseData` → full `TradeItem` aggregate. The main API does not expose a composite `/trade-items/{gln}/{number}` endpoint — it uses aspect-based subresources instead.

### Bulk endpoints

| Path | operationId | Response schema | Items schema |
|------|-------------|-----------------|--------------|
| `GET /trade-items/bulk/details` | `getBulkTradeItemDetails` | `BulkTradeItemDetailsResponse` | `TradeItemDetailsSummary` |
| `GET /trade-items/bulk/descriptions` | `getBulkTradeItemDescriptions` | `BulkTradeItemDescriptionsResponse` | `ItemDescriptionsSummary` |
| `GET /trade-items/bulk/orderings` | `getBulkTradeItemOrderings` | `BulkTradeItemOrderingsResponse` | `TradeItemOrderingsSummary` |
| `GET /trade-items/bulk/pricings` | `getBulkTradeItemPricings` | `BulkTradeItemPricingsResponse` | `TradeItemPricingSummary` |
| `GET /trade-items/bulk/relations` | `getBulkTradeItemRelations` | `BulkTradeItemRelationsResponse` | `ItemRelationSummary` |
| `GET /trade-items/bulk/logistics-details` | `getBulkTradeItemLogisticsDetails` | `BulkTradeItemLogisticsDetailsResponse` | `ItemLogisticSummary` |
| `GET /trade-items/bulk/attachments` | `getBulkTradeItemAttachments` | `BulkTradeItemAttachmentsResponse` | `ItemAttachmentSummary` |
| `GET /trade-items/bulk/enclosed-items` | `getBulkTradeItemEnclosedItems` | `BulkTradeItemEnclosedItemsResponse` | `TradeItemEnclosedSummary` |

All bulk endpoints accept query parameters: `cursor`, `limit`, `selectionId`, `supplierIdGln` (filter), `mutationDateTime`. The descriptions endpoint also accepts `language`.

## Schema Architecture

### Domain schemas (`schemas/domain/`)

| File | Purpose | Key properties |
|------|---------|----------------|
| `TradeItem.yaml` | Full trade item aggregate (all nested sections) | `supplierIdGln`, `supplierItemNumber`, `itemDetails`, `ordering`, `itemValidityDate`, `pricing[]`, `itemRelations[]`, `itemLogisticDetails[]`, `itemAttachments[]`, `packagingUnit[]`, `itemCountrySpecificFields[]` |
| `TradeItemDetails.yaml` | Merged ItemIdentification + ItemDetails (without composite key) | `manufacturerIdGln`, `manufacturerProductNumber`, `supplierAltItemNumber`, `manufacturerItemNumber`, `itemGtins[]`, `buyerItemNumber`, `discountGroupId`, `bonusGroupId`, `itemValidityDate`, `itemObsolescenceDate`, `itemStatus`, `itemCondition`, `stockItem`, `shelfLifePeriod` |
| `TradeItemDetailsSummary.yaml` | Flattened details for bulk (includes composite key) | Same as `TradeItemDetails` plus `supplierIdGln`, `supplierItemNumber` |
| `ItemDescription.yaml` | Single language description | `descriptionLanguage`, `minimalItemDescription`, `uniqueMainItemDescription`, `discountGroupDescription`, `bonusGroupDescription` |
| `ItemDescriptionsSummary.yaml` | Flattened description for bulk (includes composite key) | Composite key + all `ItemDescription` fields |
| `TradeItemOrdering.yaml` | Ordering conditions (single object per trade item) | `orderUnit`, `minimumOrderQuantity`, `orderStepSize`, `standardOrderLeadTime`, `useUnit`, `useUnitConversionFactor`, `singleUseUnitQuantity`, `alternativeUseUnit`, `alternativeUseUnitConversionFactor` |
| `TradeItemOrderingsSummary.yaml` | Flattened ordering for bulk (includes composite key) | Composite key + all `TradeItemOrdering` fields |
| `TradeItemPricing.yaml` | Single pricing entry with nested allowances/surcharges | `priceUnit`, `priceUnitFactor`, `priceQuantity`, `currencyCode`, `priceOnRequest`, `grossListPrice`, `netPrice`, `recommendedRetailPrice`, `vat`, `priceValidityDate`, `priceExpiryDate`, `allowanceSurcharges[]` |
| `TradeItemPricingsSummary.yaml` | Pricing per trade item with nested `pricings[]` for bulk | Composite key + `pricings[]` array |
| `TradeItemPricingSummary.yaml` | Flattened pricing × allowance/surcharge LEFT JOIN for bulk | Composite key + all pricing fields + all `allowanceSurcharge*` fields (nullable for rows without surcharges) |
| `AllowanceSurcharge.yaml` | Allowance or surcharge on a pricing entry | `allowanceSurchargeIndicator` (ALLOWANCE/SURCHARGE), `allowanceSurchargeType` (→ shared `AllowanceSurchargeTypes`), `allowanceSurchargeAmount`, `allowanceSurchargeSequenceNumber`, `allowanceSurchargePercentage`, `allowanceSurchargeMinimumQuantity`, `allowanceSurchargeValidityDate` |
| `ItemRelation.yaml` | Relationship to another trade item | `relatedSupplierItemNumber`, `relatedManufacturerItemNumber`, `relatedItemGtins[]`, `relationType`, `relatedItemQuantity` |
| `ItemRelationSummary.yaml` | Flattened relation for bulk (includes composite key) | Composite key + all `ItemRelation` fields |
| `ItemLogistic.yaml` | Base item logistics (dimensions, weight, volume) | `baseItemNetLength`, `baseItemNetWidth`, `baseItemNetHeight`, `baseItemNetDiameter`, `netDimensionUnit`, `baseItemNetWeight`, `netWeightUnit`, `baseItemNetVolume`, `netVolumeUnit` |
| `ItemLogisticSummary.yaml` | Flattened logistics for bulk (includes composite key) | Composite key + all `ItemLogistic` fields |
| `ItemAttachment.yaml` | Trade item attachment | `attachmentType`, `attachmentTypeSpecification`, `attachmentLanguages[]`, `attachmentFilename`, `attachmentUri`, `attachmentDescription`, `attachmentIssueDate`, `attachmentExpiryDate` |
| `ItemAttachmentSummary.yaml` | Flattened attachment for bulk (includes composite key) | Composite key + all `ItemAttachment` fields |
| `TradeItemEnclosed.yaml` | Enclosed item within a packaging unit | `enclosedSupplierItemNumber`, `enclosedManufacturerItemNumber`, `enclosedItemGtins[]`, `enclosedItemQuantity` |
| `TradeItemEnclosedSummary.yaml` | Flattened enclosed item for bulk (includes composite key + `packagingTypeCode`) | Composite key + `packagingTypeCode` + all `TradeItemEnclosed` fields |
| `PackagingUnit.yaml` | Packaging unit with logistics and enclosed items | `supplierPackagingNumber`, `manufacturerPackagingNumber`, `packagingGtins[]`, `packagingTypeCode`, `packagingUnitName`, `packagingQuantity`, `tradeItemPrimaryPackaging`, `packagingGs1Code128`, gross dimensions/weight, `tradeItemEnclosed[]` |
| `ItemCountrySpecificField.yaml` | Country-specific typed key-value fields | `csItemCharacteristicCode`, `csItemCharacteristicName`, typed values (boolean, numeric, range, string, set, select, multivalue select), `csItemCharacteristicValueUnitCode`, `csItemCharacteristicReferenceGtins[]` |
| `Supplier.yaml` | Supplier identification | `supplierName`, `supplierIdGln` |
| `ItemDetails.yaml` | ETIM-aligned ItemDetails section (used by `TradeItem.yaml` aggregate) | `itemStatus`, `itemCondition`, `stockItem`, `shelfLifePeriod`, `itemDescriptions[]` |
| `ItemIdentification.yaml` | ETIM-aligned ItemIdentification section (used by `TradeItem.yaml` aggregate) | `supplierItemNumber`, `supplierAltItemNumber`, `manufacturerItemNumber`, `itemGtins[]`, `buyerItemNumber`, `discountGroupId`, `bonusGroupId`, `itemValidityDate`, `itemObsolescenceDate` |
| `Pricing.yaml` | ETIM-aligned Pricing section (used by `TradeItem.yaml` aggregate) | `priceUnit`, `priceUnitFactor`, `priceQuantity`, `currencyCode`, `priceValidityDate`, etc. |

### Response schemas (`schemas/responses/`)

All responses follow the `data` envelope pattern:
- **Single-item**: `{ data: $ref: ./XxxResponseData.yaml }` — the `ResponseData` schema contains the composite key (`supplierIdGln` + `supplierItemNumber`) + a `$ref` to the domain schema.
- **Bulk**: `{ data: [ items: $ref: ./XxxSummary.yaml ], meta: $ref: CursorPaginationMetadata }`.

Response files: `TradeItemDetailsResponse`, `TradeItemDescriptionsResponse`, `TradeItemOrderingsResponse`, `TradeItemPricingsResponse`, `TradeItemRelationsResponse`, `TradeItemLogisticsDetailsResponse`, `TradeItemAttachmentsResponse`, `TradeItemEnclosedItemsResponse`, and their `*ResponseData` counterparts. Bulk responses: `BulkTradeItemDetailsResponse`, `BulkTradeItemDescriptionsResponse`, `BulkTradeItemOrderingsResponse`, `BulkTradeItemPricingsResponse`, `BulkTradeItemRelationsResponse`, `BulkTradeItemLogisticsDetailsResponse`, `BulkTradeItemAttachmentsResponse`, `BulkTradeItemEnclosedItemsResponse`.

## ETIM xChange V2.0 Mapping Status

### Fully mapped ETIM sections

| ETIM xChange section | API schema(s) | Notes |
|----------------------|---------------|-------|
| `ItemIdentification` + `ItemDetails` | `TradeItemDetails.yaml`, `TradeItemDetailsSummary.yaml` | Merged into single schema. Fields: `SupplierAltItemNumber`, `ManufacturerItemNumber`, `ManufacturerIdGln`, `ManufacturerProductNumber`, `ItemGtin[]`, `BuyerItemNumber`, `DiscountGroupId`, `BonusGroupId`, `ItemValidityDate`, `ItemObsolescenceDate`, `ItemStatus`, `ItemCondition`, `StockItem`, `ShelfLifePeriod` |
| `ItemDetails.ItemDescriptions[]` | `ItemDescription.yaml`, `ItemDescriptionsSummary.yaml` | Per-language: `DescriptionLanguage`, `MinimalItemDescription`, `UniqueMainItemDescription`, `DiscountGroupDescription`, `BonusGroupDescription` |
| `Ordering` | `TradeItemOrdering.yaml`, `TradeItemOrderingsSummary.yaml` | Single object per trade item: `OrderUnit`, `MinimumOrderQuantity`, `OrderStepSize`, `StandardOrderLeadTime`, `UseUnit`, `UseUnitConversionFactor`, `SingleUseUnitQuantity`, `AlternativeUseUnit`, `AlternativeUseUnitConversionFactor` |
| `Pricing[]` + `AllowanceSurcharge[]` | `TradeItemPricing.yaml`, `AllowanceSurcharge.yaml`, `TradeItemPricingSummary.yaml` | Single-item: nested `allowanceSurcharges[]` within each pricing. Bulk: LEFT JOIN flat rows (pricing × allowance/surcharge). All monetary fields use `format: decimal`. |
| `ItemRelations[]` | `ItemRelation.yaml`, `ItemRelationSummary.yaml` | All 15 relation types supported. Fields: `RelatedSupplierItemNumber`, `RelatedManufacturerItemNumber`, `RelatedItemGtin[]`, `RelationType`, `RelatedItemQuantity` |
| `ItemLogisticDetails[]` | `ItemLogistic.yaml`, `ItemLogisticSummary.yaml` | Base item net dimensions (length, width, height, diameter), weight, volume with unit codes |
| `ItemAttachments[]` | `ItemAttachment.yaml`, `ItemAttachmentSummary.yaml` | All 22 attachment types (`ATX001`–`ATX099`) + 60+ specification subtypes (`MDX*`). Fields: URI, filename, languages, description, issue/expiry dates |
| `TradeItemEnclosed[]` | `TradeItemEnclosed.yaml`, `TradeItemEnclosedSummary.yaml` | Enclosed items within packaging units: `EnclosedSupplierItemNumber`, `EnclosedManufacturerItemNumber`, `EnclosedItemGtin[]`, `EnclosedItemQuantity` |

### Partially mapped ETIM sections

- **PackagingUnit**: Schema defined (`PackagingUnit.yaml`) with identification, logistics, and enclosed items, but no bulk endpoint yet for multi-level packaging with material composition.
- **ItemCountrySpecificFields**: Full typed-value model implemented (`ItemCountrySpecificField.yaml` — boolean, numeric, range, string, set, select, multivalue select, unit code, reference GTINs) but no dedicated single-item or bulk endpoint. Used only within the `TradeItem.yaml` aggregate.
- **DiscountGroupDescription[] / BonusGroupDescription[]**: Mapped as single flattened strings within `ItemDescription.yaml`. Multilingual descriptions not available in bulk flattening (only primary language per description row).

### Intentionally excluded ETIM sections

- **ItemCountrySpecificExtensions**: Open-ended extension point, not in scope.

## Design Decisions

- TradeItem details merges ETIM's `ItemIdentification` + `ItemDetails` into a single response schema (same consolidation pattern as Product API's `ProductDetails`).
- Bulk pricings use a LEFT JOIN flat schema: each row = one pricing × allowance/surcharge combination. Rows without surcharges have all `allowanceSurcharge*` fields set to `null`.
- Ordering is a single object per trade item (not an array) — reflects the ETIM xChange structure where each trade item has exactly one `Ordering` block.
- Trade items always reference their parent product via `manufacturerIdGln` + `manufacturerProductNumber`, enabling cross-API joins.
- Descriptions are separated from details into a dedicated `/descriptions` subresource with language filtering support.
- Item descriptions include `discountGroupDescription` and `bonusGroupDescription` (denormalized from `ItemIdentification`) alongside the core description texts.
- Enclosed items are surfaced via dedicated endpoints (not nested within packaging) for simpler pagination and ETL consumption.

## Key Patterns

- All responses use the `data` envelope pattern — single-item: `data: $ref: ./XxxResponseData.yaml`, bulk: `data: [array]` + `meta: CursorPaginationMetadata`.
- Domain schemas go in `schemas/domain/`, response envelopes in `schemas/responses/`.
- Enum files are separate when reused across schemas or have >5 values.
- `format: decimal` on all ETIM-converted number fields (code-gen hint for NSwag/.NET `decimal`).
- `type: ["string", "null"]` for nullable fields (OpenAPI 3.1 / JSON Schema 2020-12).
- Every property description includes ETIM xChange traceability: `**ETIM xChange**: FieldName` + `**Path**: ...`.
- Use `examples` (plural array) in schemas, not `example` (singular, deprecated).
- Unit codes reference shared `UnitCodes.yaml` enum (order unit, price unit, use unit).
- AllowanceSurcharge types reference shared `AllowanceSurchargeTypes.yaml`.
- Tags: `TradeItems single` and `TradeItems bulk` — each endpoint has exactly one tag.
- Security: OAuth 2.0 Client Credentials with scope `read:tradeitems`.

## When Adding New Endpoints or Fields

1. Check the ETIM xChange schema at `resources/etim-xchange/ETIM xChange_Schema_V2.0-2025-11-27.json`
2. Check the mapping registry at `resources/etim-xchange/etim-field-mapping-registry.md`
3. Follow the naming conventions in `.github/copilot-instructions.md`
4. Add ETIM traceability to every new property description (`**ETIM xChange**: FieldName` + `**Path**: ...`)
5. Register all new schemas/parameters in `openapi.yaml` under `components`
6. Create `*Summary.yaml` variants for bulk endpoints (flattened with composite key)
7. Create `*ResponseData.yaml` for single-item responses (composite key + `$ref` to domain schema)
8. Regenerate the bundle after any source spec change:
   ```bash
   npx @redocly/cli bundle --config openapi/redocly.yaml tradeitem@v1 -o openapi/apis/tradeitem/generated/tradeitem-api.yaml
   ```
9. Update the mapping registry at `resources/etim-xchange/etim-field-mapping-registry.md`
10. Update `CHANGELOG.md` with the change
